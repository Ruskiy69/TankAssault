10 /*****************************************************************************
11  * Class / Struct names follow the CamelCase naming convention. They will be
12  * prefixed by Game_ if it is game-specific code. They will be prefixed by
13  * GL_ if it interfaces directly (this is critical) with OpenGL. Meaning,
14  * somewhere in the class or method definitions there has to be at least a
15  * single gl call (excluding glColor4f, because that is related to some
16  * fading effects. A perfect example of this is GL_Entity, and a perfect
17  * counter example is Game_Menu. Despite the fact that Game_Menu needs
18  * OpenGL, there's not a single direct call to it, but it is game-specific,
19  * hence the prefix.
20  *
21  * Method names follow the CamelCase naming convention just like classes
22  * and structs.
23  * Example: GL_Entity::LoadEntity(const char* p_filename);
24  *
25  * All class (not struct) member variables are preceded with m_, and mp_
26  * if it is a pointer.
27  *
28  * Function names will be declared in an lowercase underscore fashion.
29  * Example: split_string(std::string& str, char token);
30  *
31  * Complex variables such as classes and structs will be declared uppercase
32  * underscore fashion. Unless they are member variables. Then they shall be
33  * preceded with m_ and be CamelCase afterwards.
34  * Example: GL_Entity Menu_Background; MusicPlayer::m_CurrentSong;
35  *
36  * Basic variables such as integers will be lowercase underscore.
37  * Example: int my_integer;
38  *
39  * STL container (C++ standard library) variables will be named in a plural
40  * fashion with the mixed case convention regardless of what type of
41  * variables are inside. Though, if the variables inside are pointers,
42  * the container must be preceded with p_ and be plural.
43  * Example: std::vector<int> menuCodes;
44  * std::list<Tile*> p_mapTiles;
45  *
46  * STL basic variables (non-template) such as std::string will follow the
47  * same rules as normal basic variables.
48  * Example: std::ifstream& map_file;
49  *
50  * Pointers to ANY type of variable must be preceded by p_.
51  * I've spent way too much time tracking memory leaks, so I want to
52  * encourage myself to be more attentive when using pointers.
53  * Example: int* p_health = &health;
54  * GL_Entity* p_Background = NULL;
55  *
56  * Enumerations will be named in camel case, but the internal values will
57  * be named preceded by e_, be in all caps, and lack underscores, even if
58  * it's more than one "word."
59  * Example: enum ShotIndices { e_LASERSOUND, e_ROCKETSOUND };
60  *
61  * Code specific to the game is always inside of a game namespace, graphics
62  * code is in a gfx namespace, media code is in a media namespace, and
63  * in-game objects should go in the obj namespace. General code such as
64  * helper functions that do not rely in any way on external libraries
65  * should go in the namespace gk.
66  *
67  ****************************************************************************/
